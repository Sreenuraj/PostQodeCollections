# SPEC.md Format — Schema and Example

The SPEC.md is the automation contract. It must exist and be LOCKED before any workflow begins execution. Generated by `/spec-gen`. Lives at `.postqode/spec/SPEC.md`.

---

## Full Schema

```markdown
# SPEC: [Descriptive Flow Name]

## Contract Metadata
| Field | Value |
|---|---|
| Created | [date] |
| Status | DRAFT → LOCKED (on user approval) |
| Domain | web-automation |
| Framework | [name] / TBD |
| Language | [name] / TBD |
| Target URL | [full URL] |
| Viewport | [e.g. 1280x800] |
| Spec Author | Strategist persona via /spec-gen |

## What & Why
[2-3 sentences: what user flow is being automated and why it needs automation]

## Scope
**IN scope:**
- [exact list of what will be automated]

**OUT of scope:**
- [explicit list of what will NOT be touched — prevents scope creep]

## Step Definitions
| # | Step | Action | Input | Expected Outcome | Component |
|---|---|---|---|---|---|
| 1 | Navigate to app | Open browser, go to URL | TARGET_URL | Login page is visible | — |
| 2 | Enter credentials | Fill username and password fields | user: X, pass: Y | Fields populated | login-form |
| 3 | Submit login | Click login/submit button | — | Dashboard page loads | login-form |
| 4 | ... | ... | ... | ... | ... |

## Success Criteria (Reviewer Rubric Reference)
- [ ] Every step in Step Definitions has corresponding test code
- [ ] All steps pass headless validation with zero retries
- [ ] No arbitrary sleep() or fixed-time waits in generated code
- [ ] Every locator has ≥1 fallback strategy
- [ ] Every major action (navigate, submit, drag) has a DOM assertion after it
- [ ] TIP evidence cited in code comments for each step

## Anti-Patterns (FORBIDDEN in generated code)
- Never use sleep() or waitForTimeout() with a fixed time
- Never batch checklist rows
- Never hardcode credentials or environment URLs inline
- Never use auto-generated CSS class names as primary locators
- Never write code until snapshot evidence is captured for that step

## Validation Gate
| Field | Value |
|---|---|
| Command | [exact test run command] |
| Mode | headless |
| Retries | 0 |
| Expected | All assertions pass, no console errors |
```

---

## Status Lifecycle

```
DRAFT   → Agent-generated; not yet user-approved
LOCKED  → User-approved; safe for execution to begin
```

**LOCKED does not mean immutable.** New steps can be added mid-session by the Strategist persona:
1. Strategist opens SPEC.md, adds step to Step Definitions table
2. Strategist adds step to appropriate `pending-groups/*.md` file  
3. Brief user gate: "Added Step [N] to SPEC.md. Will execute in Group [X]. (A) Continue  (B) Adjust"
4. NO full re-plan unless user explicitly requests it

## Step Definition Rules

### Cohesive Interaction Heuristic (Step Sizing)
Do not blindly map one user action (click/type) to one step in the spec. Group logically related actions within the same component into cohesive steps. This prevents checklist bloat and speeds up execution.

**Rule of Thumb:** If actions happen immediately together on the same component to achieve one micro-goal, box them into one step.

❌ **Inefficient (Too granular):**
- Step N: Fill email field
- Step N+1: Fill password field
- Step N+2: Click submit button

✅ **Efficient (Cohesive step):**
- Step N: Fill login credentials and submit

❌ **Too Broad (Needs decomposition):**
- Step N: Complete the checkout process and pay

✅ **Efficient Decomposition:**
- Step N: Fill shipping address
- Step N+1: Select payment method and enter card details
- Step N+2: Click place order and assert confirmation

### Expected Outcome Must Be Observable
❌ Bad: "The action completes"
✅ Good: "A success toast appears with text 'Vote submitted'"
✅ Good: "URL changes to /dashboard"
✅ Good: "The data grid shows a new row with name 'Alpha'"

### Component Column
The `Component` column names the logical UI block containing the target element. This drives element map creation during exploration:
- `login-form` → `element-maps/login-page__login-form.json`
- `vote-slider` → `element-maps/vote-page__vote-slider.json`
- Leave blank if the interaction has no distinct component (e.g., page navigation)

## Minimal Working Example

```markdown
# SPEC: Proxy Vote Submission Flow

## Contract Metadata
| Field | Value |
|---|---|
| Created | 2026-03-29 |
| Status | LOCKED |
| Domain | web-automation |
| Framework | TBD |
| Language | TBD |
| Target URL | https://vote.example.com |
| Viewport | 1280x800 |

## What & Why
Automates the end-to-end proxy voting flow where a shareholder logs in, navigates to an active meeting, sets their vote allocation via a slider, and submits. Manual testing is error-prone for the slider interaction.

## Scope
**IN scope:** Login, meeting navigation, slider adjustment, vote submission, confirmation
**OUT of scope:** Admin panel, report generation, email notifications

## Step Definitions
| # | Step | Action | Input | Expected Outcome | Component |
|---|---|---|---|---|---|
| 1 | Navigate | Open browser | https://vote.example.com | Login screen visible | — |
| 2 | Login | Fill + submit credentials | user: test@co.com, pass: Test123! | Dashboard visible | login-form |
| 3 | Select meeting | Click active meeting | "Q1 2026 Annual Meeting" | Meeting detail page loads | meeting-list |
| 4 | Set vote % | Drag slider to 60% | 60 | Slider displays "60%" | vote-slider |
| 5 | Submit vote | Click "Submit Vote" | — | Confirmation modal appears | vote-submit |
| 6 | Confirm | Click "Confirm" | — | Success message: "Vote recorded" | vote-submit |

## Success Criteria
- [ ] All 6 steps pass headless validation
- [ ] Slider assertion accepts 59-61% range (±1 tolerance)
- [ ] Vote submission confirmed via API response AND UI message

## Anti-Patterns
- Never hardcode the meeting name — read from env config
- Slider: use JS value setter first, track click as fallback

## Validation Gate
| Field | Value |
|---|---|
| Command | TBD (set during Setup) |
| Mode | headless, 0 retries |
```
